## What this is

Warden is a Bash CLI that orchestrates Docker-based PHP developer environments (Magento 1/2, Laravel, Symfony, Drupal, Shopware, etc.). It is **pure Bash + docker-compose YAML** — there is no build step, no compiler, no package manager, and no test suite in this repo. You edit shell scripts and compose files directly; changes take effect on the next `warden` invocation.

The `version` file contains `in-dev` on the development branch; release tags set it to a real version.

## Running / developing

```bash
bin/warden <command> [args]      # run the CLI directly from the repo
bin/warden svc up                # start global shared services (traefik, dnsmasq, tunnel, mailpit, ...)
bin/warden env up                # start the current project's environment (run from inside a project dir)
```

There is no lint or test command wired up. If validating shell changes, run `shellcheck` manually on the edited `*.cmd` / `utils/*.sh` files (the code uses `# shellcheck disable=...` directives, so shellcheck is the expected linter).

`bin/warden install` (invoked automatically via `assertWardenInstall` on `svc` commands) seeds `~/.warden` and appends a `tunnel.warden.test` block to `/etc/ssh/ssh_config` (needs sudo).

## Architecture

### Dispatch (`bin/warden`)
The entrypoint resolves `$WARDEN_DIR` (following symlinks), sources `utils/{core,svc,env}.sh`, validates docker + docker-compose ≥ 2.2.3, then resolves the subcommand to a `*.cmd` file and **`source`s it** (commands are sourced, not exec'd — they share the entrypoint's environment and functions). Command lookup is **override-ordered**, first match wins:
1. `<project>/.warden/commands/<cmd>.cmd` (per-project)
2. `~/.warden/commands/<cmd>.cmd` (per-user)
3. `commands/<cmd>.cmd` (this repo, the defaults)

Each command has a paired `<cmd>.help` file; `-h`/`--help` sources `commands/usage.cmd`. Commands in `WARDEN_CMD_ANYARGS` (db, debug, doctor, env, redis, shell, svc, sync) pass all flags through to the underlying tool instead of having warden parse them.

### Two orchestration layers
- **`svc` (global, shared):** one `warden` project spanning all environments. Reverse proxy (Traefik) for SSL/routing, dnsmasq for `*.test` DNS, an SSH tunnel for DB access, mailpit, optional portainer/phpMyAdmin. Defined in `docker/docker-compose.*.yml`, run with `--project-directory ~/.warden -p warden`.
- **`env` (per-project):** one docker-compose project per developer project, keyed by `WARDEN_ENV_NAME`. `env.cmd` assembles the compose file list and runs `docker compose --project-directory <project> -p <env-name>`.

The two layers connect via **peered services**: `connectPeeredServices` / `disconnectPeeredServices` (in `utils/core.sh`) attach the global Traefik/tunnel/mailhog/phpmyadmin containers onto each project's docker network so the shared proxy can route into project containers.

### Compose-file layering (the core mechanism)
`env.cmd` builds a `DOCKER_COMPOSE_ARGS` array of `-f <file>` pairs by calling `appendEnvPartialIfExists <name>` (in `utils/env.sh`) for each service the project's `WARDEN_*` flags enable (nginx, db, redis, varnish, elasticsearch, rabbitmq, blackfire, selenium, ...). For each partial it searches **6 base paths × suffixes**, appending every file that exists (later files override earlier ones):

- Base paths (override order): `environments/includes`, `environments/<type>`, then the same two under `~/.warden/environments`, then under `<project>/.warden/environments`.
- Suffixes: `.base.yml`, `.<darwin|linux>.yml`, and `.mutagen_compose.yml` (only when mutagen is enabled).

This is how a project can override or extend any service definition by dropping a file in its `.warden/environments/`. Environment **types** live in `environments/<type>/` (each needs a `<type>.base.yml`); shared service partials live in `environments/includes/`. `init.env` files seed a project's `.env` on `env-init`.

### Configuration / env vars
`loadEnvConfig` (in `utils/env.sh`) walks up from CWD to find a project `.env` (must contain `WARDEN_ENV_NAME` + `WARDEN_ENV_TYPE`), then loads it. Behavior is driven almost entirely by `WARDEN_*` env vars read from the project `.env` and `~/.warden/.env` (e.g. `WARDEN_DB`, `WARDEN_REDIS`, `WARDEN_VARNISH`, `WARDEN_MUTAGEN_ENABLE`, `WARDEN_PHPMYADMIN_ENABLE`). `OSTYPE` selects the `darwin`/`linux` subtype; mutagen file-sync defaults on for macOS, off for Linux.

**Security — loading project `.env`:** `loadEnvFile` parses dotenv lines manually (regex-validated keys, prefix-filtered, quotes stripped but contents never expanded). This deliberately replaced an `eval "$(grep ...)"` pattern that was RCE — **do not reintroduce `eval` for parsing untrusted project `.env` files.** Note `eval` is still used in a few `*.cmd` files against the trusted `~/.warden/.env` only.

## Conventions when adding code

- Every `*.cmd` / `utils/*.sh` starts with the guard `[[ ! ${WARDEN_DIR} ]] && ... exit 1` — these are meant to be sourced, never run directly.
- A new subcommand = add `commands/<name>.cmd` + `commands/<name>.help`. If it should receive arbitrary pass-through flags, add it to `WARDEN_CMD_ANYARGS` in `bin/warden`.
- A new optional service = add `environments/includes/<name>.base.yml` (+ OS-specific variants), gate it behind a `WARDEN_<NAME>` flag with an `appendEnvPartialIfExists` call in `commands/env.cmd`. If it needs network peering, handle it in `getPeeredServices`.
- User-facing messages use the `warning` / `error` / `fatal` helpers in `utils/core.sh` (colored stderr).
- Update `CHANGELOG.md` for user-visible changes.
